Introduction
Prometheus main purpose is to collect metrics. Metrics are one of the core components for Observability which is quite important for any application developed today.
What is a metric?
•Metrics are values that allow us to understand how systems and business processes are working, these values usually allow us to understand what may be happening. For example CPU usage, Memory usage, Exception Count, Request Duration, Number of items sold, etc...
What are the types of metrics supported by Prometheus?
•Counter - Counter metrics are used for measurements that only increase.
•Gauge - Gauge metrics are used for measurements that can arbitrarily increase or decrease.
•Histogram - Histogram metrics are useful to represent a distribution of measurements. They are often used to measure request duration or response size.
oHistograms divide the entire range of measurements into a set of intervals???named buckets???and count how many measurements fall into each bucket.
•Summaries - Similar to histograms
oSummaries include a counter with the total number of measurements. The metric name uses the _count suffix.
oSummaries also include a counter with the sum of the values of all measurements. The metric name uses the _sum suffix.

How are metrics collected?
•Metrics are exposed by applications through an endpoint and Prometheus scraps the endpoints using pull HTTP requests.
•The endpoint scrapped by Prometheus is usually "/metrics".
•Applications can push metrics via an intermediary gateway but this should only be used in particular scenarios.
How can we configure Prometheus metrics collection internal?
•We configure Prometheus metrics collection interval by defining a scrape_internal properly in Prometheus config file when it gets deployed.
How can I access the metrics collected?
•Metrics can be queried using PROMQL which is a query language used by Prometheus.
•Metrics can also be exposed into a dashboard such as Grafana for example.
What are targets?
•Targets are the systems Prometheus monitors and extracts metrics from (examples: Linux Server, Single Page Application, Apache Server, Database)
Can I extended Prometheus behaviour to collect metrics from other endpoints?
•Yes we can achieve this by defining our own jobs. Configuring extra jobs allow us to extend the collection of metrics to other endpoints.
Architecture
From an architectural point of view Prometheus comprises the following components:
•Prometheus server which is used to scrap and store time-series data. It also provides a user interface to query the metrics using PROM QL.
•Client libraries used to or instrumenting application code.
•Push Gateways usually support metrics collection from short-lived jobs.
•Exporters for services that cannot be directly instrumented by Prometheus.
•Alert manager to handle real-time alerts based on triggers defined.
How can I use Client Libraries?
•These are usually installed in the source code by developers and once properly configured will expose the /metrics endpoint that Prometheus uses to scrap metrics.
•These libraries are available for multiple languages.
How exporters work?
•An export will fetch metrics from a system that cannot expose /metrics endpoint and will convert those metrics into Prometheus format, these metrics converted to Prometheus format will then be exposed by the exporter in a /metrics endpoint that is then consumed by Prometheus.
•Usually exports are available as docker images.
Prometheus and Kubernetes
•Prometheus and Kubernetes are compatible and by default when Prometheus is deployed into Kubernetes it will be able to collect Kubernetes metrics from the Nodes (Virtual Machines that provide resources to the cluster) that are a part of the cluster.
When I deploy Prometheus operator in Kubernetes what is monitored by default?
•By default you will get your worker nodes monitored alongside with Kubernetes components such as PODs, Deployments, Stateful Sets.
When I deploy Prometheus in Kubernetes using the operator I see a kube-state-metrics POD what is it?
•Kube state metrics is used to scrape metrics from components, pod, deployments, stateful sets. This POD essentially provides a part of the out of the box Kubernetes monitoring functionality.
When I deploy Prometheus in Kubernetes using the operator I see a prometheus-node-exporter POD what is it?
•The node exporter metrics is used to extract metrics from every worker node that is a part of Kubernetes. This essentially extracts metrics about CPU Usage, Load Usage and converts these metrics to a Prometheus compatible format.?? Prometheus Operator To Deploy In Kubernetes
•An operator is essentially a Kubernetes concept, essentially Kubernetes contains a lot of different concepts, such as PODs, Deployments, Services, Ingress and others. The Prometheus operator makes the deployment of Prometheus simple as all configuration required by Prometheus Stack come via Custom Resource Definitions (CRDs).
•Fundamentally the Prometheus operator will automate and manage the Prometheus monitoring stack that runs in a Cluster. The Prometheus operator takes advantage of monitor object to dynamically discover endpoints and scrape metrics without us having to update Prometheus configurations.

My PODs are not being monitor but they are exposing "/metrics" endpoint?
•Prometheus Operators do not support annotation based discovery of PODs, instead it uses PodMonitors and ServiceMonitors.
What are Service Monitors and Pod Monitors?
•ServiceMonitor and PodMonitor describe the targets to be monitored by Prometheus.
oServiceMonitor declaratively specifies how groups of Kubernetes services should be monitored. When a ServiceMonitor is created, the Prometheus Operator updates the Prometheus scrape configuration to include the ServiceMonitor configuration. Then Prometheus begins scraping metrics from the endpoint defined in the ServiceMonitor.
oAny Services in your cluster that match the labels located within the ServiceMonitor selector field will be monitored based on the endpoints specified on the ServiceMonitor.
oPodMonitor is created, the Prometheus Operator updates the Prometheus scrape configuration to include the PodMonitor configuration. Then Prometheus begins scraping metrics from the endpoint defined in the PodMonitor. Any Pods in your cluster that match the labels located within the PodMonitor selector field will be monitored based on the podMetricsEndpoints specified on the PodMonitor. For more information on what fields can be specified, please look at the

What is annotation based discovery of services (NOT SUPPORTED BY PROMETHEUS OPERATOR)?
•Annotation based discovery of services consists in adding annotations to the PODs so that they can be discovered by the Prometheus Server, these annotations are as follows:
o"prometheus.io/scrape":"true"
o"prometheus.io/path":"/metrics"
o"prometheus.io/port":"80"
Prometheus Downsides
•Hard to scale
As servers are stand-alone and self-contained it is hard to manage infrastructure that will have the capacity to aggregate all the metrics collected.
•Limits Monitoring
Usually the use of a single Prometheus node allow us to tackle the issues with scaling however it limits the amount of metrics we can collect. In order to minimise these issues we need to limited the metrics being collected by the single instance or vertically scale the instance by providing it more resources.
•Metrics are not collected in real time there is a small latency between the time values are generated and when they are scraped.
Prometheus Storage
•Data collected by Prometheus is stored on Disk but it is also possible to integrate with databases, however the data follows its own format which may require some adjustments in order to ingest it into a SQL database.
